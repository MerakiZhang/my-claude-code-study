# 03 - 运行时代码检测：让程序"认得出"自己在哪运行

## 学习目标

读完本节后，你将能够：
1. 理解为什么同一份代码需要在不同运行时（Bun / Node.js）下运行
2. 掌握两种运行时检测方法：`process.versions.bun` 和 `typeof Bun`
3. 写出兼容多运行时的健壮代码
4. 理解 Claude Code 中 `isRunningWithBun()` 和 `isInBundledMode()` 的设计意图

---

## 3.1 为什么需要"运行时检测"？

想象你是一个演员，今天在上海演出，明天在北京演出。两场演出的舞台（运行时）不同，但你的剧本（代码）是同一份。你需要在演出时"看一看"自己在哪个舞台，然后决定用哪个道具。

在 Claude Code 的源码中，同一份代码面临三种运行场景：

| 场景 | 说明 |
|------|------|
| **开发环境** | 开发者用 `bun run src/main.tsx` 直接运行源码 |
| **Bun 编译的独立可执行文件** | `bun build --compile` 生成的单个二进制文件，用户直接双击运行 |
| **Node.js 环境** | 某些企业环境或 CI/CD 流水线可能只能用 Node.js |

因此，Claude Code 的代码必须"聪明"到能自动判断自己在哪种环境下，然后做出正确的选择。

## 3.2 检测方法一：`process.versions.bun`

### 原理

每个 JavaScript 运行时都会在 `process.versions` 对象中留下自己的"签名"。

- 在 **Bun** 中运行：
  ```typescript
  console.log(process.versions.bun); // 输出 "1.2.0"（或你的 Bun 版本号）
  console.log(process.versions.node); // 也可能有，表示兼容的 Node 版本
  ```

- 在 **Node.js** 中运行：
  ```typescript
  console.log(process.versions.bun); // 输出 undefined
  console.log(process.versions.node); // 输出 "22.0.0"（或你的 Node 版本号）
  ```

### 动手实验

创建 `detect-runtime.ts`：

```typescript
// detect-runtime.ts

console.log("=== 运行时检测报告 ===");
console.log(`process.versions.bun: ${process.versions.bun}`);
console.log(`process.versions.node: ${process.versions.node}`);

if (process.versions.bun !== undefined) {
  console.log("✅ 当前运行在 Bun 环境中！");
  console.log(`Bun 版本: ${process.versions.bun}`);
} else {
  console.log("❌ 当前不在 Bun 环境中（可能是 Node.js）");
}
```

分别用 Bun 和 Node.js 运行：

```bash
# 用 Bun 运行
bun detect-runtime.ts

# 用 Node.js 运行（如果你安装了 Node.js）
node detect-runtime.ts
# 注意：如果文件里有 TypeScript 语法，Node.js 可能报错。
# 这是正常的，正好说明了为什么需要检测！
```

### Claude Code 中的实际应用

在 `src/utils/bundledMode.ts` 中，有一个非常简洁的检测函数：

```typescript
/**
 * 检测当前运行时是否为 Bun。
 * 在以下情况返回 true：
 * - 用 `bun` 命令直接运行 JS/TS 文件
 * - 运行 Bun 编译的独立可执行文件
 */
export function isRunningWithBun(): boolean {
  // https://bun.com/guides/util/detect-bun
  return process.versions.bun !== undefined;
}
```

这就是源码中最核心的 Bun 检测逻辑。**只有 3 行代码，却撑起了整个项目的运行时分支决策。**

## 3.3 检测方法二：`typeof Bun !== 'undefined'`

### 原理

Bun 在全局作用域中定义了一个名为 `Bun` 的对象，包含各种专属 API（如 `Bun.hash`、`Bun.write` 等）。

- 在 **Bun** 中：`typeof Bun` 是 `"object"`
- 在 **Node.js** 中：`typeof Bun` 是 `"undefined"`（因为不存在这个变量）

### 动手实验

创建 `detect-bun-object.ts`：

```typescript
// detect-bun-object.ts

console.log("=== Bun 全局对象检测 ===");
console.log(`typeof Bun: ${typeof Bun}`);

if (typeof Bun !== 'undefined') {
  console.log("✅ Bun 全局对象存在！");
  console.log(`Bun 版本: ${Bun.version}`);
} else {
  console.log("❌ Bun 全局对象不存在");
}
```

运行：

```bash
bun detect-bun-object.ts
```

### 两种方法的区别

| 检测方式 | 检测目标 | 使用场景 |
|---------|---------|---------|
| `process.versions.bun` | 是否在 Bun 运行时 | 判断运行环境版本、选择包管理器命令 |
| `typeof Bun !== 'undefined'` | 能否使用 Bun 专属 API | 调用 `Bun.hash`、`Bun.write` 等之前做安全检查 |

> 💡 **最佳实践**：在调用 Bun 专属 API 之前，先用 `typeof Bun !== 'undefined'` 检查；在需要判断版本号时，用 `process.versions.bun`。

## 3.4 写出兼容多运行时的代码

### 模式：优先使用 Bun API，回退到 Node.js

这是 Claude Code 源码中最常见的模式。看一个简化版的例子：

```typescript
// hash-utils.ts

import { createHash } from 'crypto'; // Node.js 内置模块

/**
 * 计算字符串的哈希值
 * 在 Bun 下用 Bun.hash（快约 10 倍）
 * 在 Node.js 下用 crypto.createHash
 */
export function hashContent(content: string): string {
  // 第一步：检查 Bun 全局对象是否存在
  if (typeof Bun !== 'undefined') {
    // Bun.hash 返回一个 number，转成字符串
    return Bun.hash(content).toString();
  }

  // 第二步：回退到 Node.js 的方式
  return createHash('sha256').update(content).digest('hex');
}

// 测试
console.log(hashContent("Hello, Bun!"));
```

### 模式：根据运行时调整行为

在 `src/main.tsx` 中，`isBeingDebugged()` 函数对 Bun 和 Node.js 的调试参数做了不同处理：

```typescript
function isBeingDebugged() {
  const isBun = isRunningWithBun();

  // 检查进程参数中是否包含 inspect 调试标志
  const hasInspectArg = process.execArgv.some(arg => {
    if (isBun) {
      // Bun 不支持 Node.js 遗留的 --debug 和 --debug-brk 标志
      return /--inspect(-brk)?/.test(arg);
    } else {
      // Node.js 支持 --inspect 和遗留的 --debug 标志
      return /--inspect(-brk)?|--debug(-brk)?/.test(arg);
    }
  });

  return hasInspectArg;
}
```

> 💡 **为什么要有这个区别？** Bun 和 Node.js 的调试协议不同。Node.js 有遗留的 `--debug` 标志（现已废弃），而 Bun 从来没有支持过这些遗留标志。

## 3.5 更复杂的版本检测

有时候你不仅需要知道"是不是 Bun"，还需要知道"Bun 的版本够不够新"。

### 场景：Windows 终端 VT 模式支持

在 `src/keybindings/defaultBindings.ts` 中：

```typescript
import { satisfies } from 'semver';

const SUPPORTS_TERMINAL_VT_MODE =
  getPlatform() !== 'windows' ||
  (isRunningWithBun()
    ? satisfies(process.versions.bun, '>=1.2.23')
    : satisfies(process.versions.node, '>=22.17.0 <23.0.0 || >=24.2.0'));
```

这段代码的意思是：
- 如果不是 Windows，直接支持
- 如果是 Windows + Bun：要求 Bun 版本 >= 1.2.23
- 如果是 Windows + Node.js：要求 Node 版本 >= 22.17.0 或 >= 24.2.0

### 动手实验

创建 `version-check.ts`：

```typescript
// version-check.ts

function checkVersion(): void {
  if (process.versions.bun) {
    const version = process.versions.bun;
    const [major, minor, patch] = version.split('.').map(Number);
    console.log(`Bun 版本: ${version}`);
    console.log(`主版本: ${major}, 次版本: ${minor}, 补丁: ${patch}`);

    if (major > 1 || (major === 1 && minor >= 2)) {
      console.log("✅ 版本满足 >= 1.2.0 的要求");
    } else {
      console.log("❌ 版本太旧，请升级 Bun");
    }
  } else {
    console.log("当前不是 Bun 运行时");
  }
}

checkVersion();
```

## 3.6 `isInBundledMode()` —— 检测是否为独立可执行文件

这是 Claude Code 特有的一个重要概念。

### 什么是 "Bundled Mode"？

当你用 `bun build --compile` 把项目打包成一个单独的可执行文件时，这个文件内部可以"嵌入"其他文件（如配置文件、二进制工具等）。

在源码中，通过 `Bun.embeddedFiles` 可以检测是否存在这些嵌入的文件：

```typescript
// src/utils/bundledMode.ts

/**
 * 检测当前是否以 Bun 编译的独立可执行文件运行。
 * 通过检查是否存在嵌入的文件来判断。
 */
export function isInBundledMode(): boolean {
  return (
    typeof Bun !== 'undefined' &&
    Array.isArray(Bun.embeddedFiles) &&
    Bun.embeddedFiles.length > 0
  );
}
```

### 为什么这很重要？

想象 Claude Code 发布给用户的是一个单独的文件 `claude.exe`：

- **开发时**：`bun run src/main.tsx`，此时 `isInBundledMode()` 返回 `false`
- **发布后**：用户双击 `claude.exe`，此时 `isInBundledMode()` 返回 `true`

源码中很多逻辑都依赖这个判断：
- 开发时加载外部的 `ripgrep` 工具；发布后使用内嵌的版本
- 开发时通过 `node_modules` 加载依赖；发布后所有代码都在一个文件里
- 开发时显示 npm 弃用警告；发布后跳过这些警告

### 动手实验

创建 `detect-bundled.ts`：

```typescript
// detect-bundled.ts

function checkBundledMode(): void {
  if (typeof Bun === 'undefined') {
    console.log("❌ 不是 Bun 运行时，不可能是 bundled mode");
    return;
  }

  console.log(`Bun.embeddedFiles 类型: ${typeof Bun.embeddedFiles}`);

  if (Array.isArray(Bun.embeddedFiles) && Bun.embeddedFiles.length > 0) {
    console.log("✅ 当前运行在 bundled mode（独立可执行文件）");
    console.log(`嵌入文件数量: ${Bun.embeddedFiles.length}`);
    console.log("嵌入文件列表:");
    for (const file of Bun.embeddedFiles) {
      console.log(`  - ${file.name}`);
    }
  } else {
    console.log("ℹ️ 当前不是 bundled mode（普通源码运行）");
  }
}

checkBundledMode();
```

用普通方式运行：

```bash
bun detect-bundled.ts
# 输出：当前不是 bundled mode（普通源码运行）
```

## 3.7 回退策略（Fallback）的设计模式

在 Claude Code 中，几乎所有 Bun 专属 API 都有 Node.js 的"备胎"。这是一个非常重要的工程思想：

```typescript
// 模式模板
function doSomething(input: string): Result {
  // 方案 A：用 Bun 的高速公路（如果有的话）
  if (typeof Bun !== 'undefined') {
    return Bun.somethingFast(input);
  }

  // 方案 B：用 Node.js 的普通公路（保证兼容）
  return nodeJsAlternative(input);
}
```

### 更多源码实例

| 功能 | Bun API | Node.js 回退 | 文件 |
|------|---------|-------------|------|
| 哈希 | `Bun.hash()` | `crypto.createHash()` | `src/utils/hash.ts` |
| 查找命令 | `Bun.which()` | 子进程执行 `which`/`where` | `src/utils/which.ts` |
| YAML 解析 | `Bun.YAML.parse()` | npm `yaml` 包 | `src/utils/yaml.ts` |
| 语义版本 | `Bun.semver` | npm `semver` 包 | `src/utils/semver.ts` |
| 终端宽度 | `Bun.stringWidth()` | `string-width` npm 包 | `src/ink/stringWidth.ts` |

## 小结

| 检测目标 | 代码 | 返回值含义 |
|---------|------|-----------|
| 是否是 Bun 运行时 | `process.versions.bun !== undefined` | `true` = 是 Bun |
| Bun 版本号 | `process.versions.bun` | 字符串，如 `"1.2.0"` |
| Bun 全局对象是否存在 | `typeof Bun !== 'undefined'` | `true` = 可以用 Bun API |
| 是否是独立可执行文件 | `Bun.embeddedFiles?.length > 0` | `true` = `bun build --compile` 产物 |

| 设计模式 | 说明 |
|---------|------|
| 运行时检测 | 用条件判断区分 Bun / Node.js |
| 回退策略 | Bun API 不可用时，使用 Node.js 替代方案 |
| 版本检查 | 某些功能需要特定版本的 Bun |

## 练习题

1. 编写 `runtime-info.ts`，打印当前运行时的完整信息（是否是 Bun、版本号、是否是 bundled mode）。
2. 编写 `safe-hash.ts`，实现一个 `safeHash(input: string): string` 函数：在 Bun 下用 `Bun.hash`，在 Node.js 下用 `crypto.createHash('sha256')`。
3. （思考题）为什么 `isInBundledMode()` 需要检查 `typeof Bun !== 'undefined'` 和 `Array.isArray(Bun.embeddedFiles)` 两个条件？只检查一个行不行？

> 参考答案：需要双重保险。`typeof Bun !== 'undefined'` 确保运行时是 Bun；`Array.isArray(Bun.embeddedFiles)` 确保 `embeddedFiles` 属性存在且是数组（防止未来 Bun 版本改变这个 API 的形状）。在 Node.js 下，`Bun` 对象不存在，如果直接访问 `Bun.embeddedFiles.length` 会抛出 `ReferenceError`。
