# 第四章：模块系统

> 本章目标：理解为什么需要把代码拆分成多个文件，以及如何正确地导入和导出代码。
>
> 想象你在整理房间：如果把所有东西堆在一个箱子里，找起来会很困难。分成不同的盒子——衣服一盒、书籍一盒、电子产品一盒——就清晰多了。模块系统就是代码的"整理盒子"。

---

## 4.1 为什么需要模块？

Claude Code 这个项目有 **上千个文件**。如果所有代码都写在一个文件里：

- 打开文件要等很久
- 找一个函数要翻几千行
- 多人合作时容易互相冲突
- 改一个地方可能不小心影响到其他地方

**模块**让你可以把代码拆分成多个文件，每个文件只负责一件事，并且只暴露需要暴露的内容。

> 💡 **类比**：模块就像乐高积木——每个积木块是独立的，但可以通过标准接口拼在一起，搭出复杂的作品。

---

## 4.2 ES Module 基础

现代 JavaScript 使用 **ES Module**（简称 ESM）作为标准模块系统。

### export（命名导出）

一个文件默认对外什么都不暴露。你需要显式地用 `export` 关键字标记哪些内容可以被别人使用。

```js
// math.js
export const PI = 3.14159;

export function add(a, b) {
  return a + b;
}

export function subtract(a, b) {
  return a - b;
}

// 没有 export 的变量，其他文件无法访问
const secret = '这是内部使用的变量';
```

> 💡 **类比**：`export` 就像店铺的橱窗——你把商品摆出来给顾客看，但仓库里的存货不外露。

### import（命名导入）

另一个文件想使用 `math.js` 的内容，需要用 `import`：

```js
// main.js
import { PI, add, subtract } from './math.js';

console.log(PI);           // 3.14159
console.log(add(2, 3));    // 5
console.log(subtract(5, 2)); // 3
```

注意几个细节：

1. **花括号 `{}`**：命名导入需要用花括号包裹你要导入的名字
2. **`from './math.js'`**：指定模块的路径
3. **`.js` 后缀**：在 Node.js 的 ESM 模式下，导入路径**必须写 `.js` 后缀**（即使是导入一个 `.ts` 文件，路径也要写 `.js`）

### 源码例子：Task.js 中的导出

在 Claude Code 的 `src/Task.js` 中（概念转换为纯 JavaScript）：

```js
// src/Task.js
export const TASK_STATUSES = ['pending', 'running', 'completed', 'failed', 'killed'];

export function isTerminalTaskStatus(status) {
  return status === 'completed' || status === 'failed' || status === 'killed';
}

// 没有 export，外部看不到
function generateTaskId() {
  return 'task_' + Math.random().toString(36).slice(2);
}
```

另一个文件可以这样导入：

```js
// src/someOtherFile.js
import { TASK_STATUSES, isTerminalTaskStatus } from './Task.js';

console.log(TASK_STATUSES);
console.log(isTerminalTaskStatus('completed')); // true
```

## 4.3 默认导出（default export）

一个文件可以有且仅有一个"默认导出"：

```js
// greet.js
export default function greet(name) {
  return '你好，' + name;
}
```

导入时不需要花括号，名字可以随便起：

```js
// main.js
import greet from './greet.js';
import sayHello from './greet.js';  // 同一个模块，换个名字也行

console.log(greet('小明'));      // 你好，小明
console.log(sayHello('小红'));   // 你好，小红
```

也可以导出一个值或对象：

```js
// config.js
export default {
  apiUrl: 'https://api.example.com',
  timeout: 5000,
};

// main.js
import config from './config.js';
console.log(config.apiUrl); // https://api.example.com
```

### Claude Code 为什么很少用默认导出？

在 Claude Code 的源码中，**极少使用 `export default`**。团队更偏好**命名导出**，因为：

1. **IDE 中更容易搜索和跳转**——你想找一个函数，直接搜名字就能找到定义
2. **重命名时更安全**——IDE 可以自动重构所有用到的地方
3. **导入时必须用正确的名字**——不容易混淆或拼错

> 🎯 **建议**：学习 Claude Code 的风格，尽量用命名导出。默认导出只在确实只有一个主要导出时才用（比如一个配置文件、一个类）。

## 4.4 通配符导入（Namespace Import）

有时候你想导入一个模块的**所有导出**，放在一个"命名空间"对象里：

```js
import * as math from './math.js';

console.log(math.PI);          // 3.14159
console.log(math.add(2, 3));   // 5
```

> 💡 **类比**：`* as math` 就像把 `math.js` 这个工具箱整个搬过来，使用时说 `math.工具名`。

### 源码例子：React 的导入

在 Claude Code 的 TSX 文件中，这种写法很常见：

```js
// 来自 Claude Code 的 TSX 文件（转换为纯 JavaScript）
import * as React from 'react';

// 使用时
const element = React.createElement('div', null, 'Hello');
```

这里把 `react` 模块的所有导出都放到 `React` 这个对象里，然后通过 `React.createElement`、`React.useState` 等方式使用。

当然，也可以直接按需导入你需要的函数：

```js
import { useState, useEffect } from 'react';
```

## 4.5 TypeScript 中的类型导入与导出（编译时概念）

在纯 JavaScript 中，所有 `import` 导进来的都是**运行时的值**（函数、变量、对象等）。但 TypeScript 为了支持静态类型检查，增加了一种特殊的导入导出语法：

```ts
// 只导入类型信息（编译后会被删除）
import type { TaskStatus } from './Task.js';

// 只导出类型信息（编译后会被删除）
export type { TaskStatus } from './Task.js';
```

`import type` 和 `export type` 只在编译时存在，用来告诉 TypeScript 编译器"这里用到的是某个类型"。它们编译成 JavaScript 后会被**完全擦除**，不会生成任何运行时代码，也不会影响程序的执行。

TypeScript 还支持把类型和非类型混在一起导入或重导出：

```ts
// 混合导入：bar 是运行时值，Foo 是纯类型
import { type Foo, bar } from './module.js';

// 混合重导出
export { type Foo, bar } from './module.js';
```

> 💡 **底层原理**：JavaScript 是动态类型语言，变量和函数在运行时没有固定的类型标签。TypeScript 在编译阶段进行类型检查，检查完成后会把所有类型相关的代码（包括类型注解、`import type`、`interface`、`type` 别名等）全部擦除，输出纯粹的 JavaScript。因此，`import type` 在 JavaScript 中没有对应的机制——如果你写纯 JS，这个概念根本不存在。

当你阅读 Claude Code 的 TypeScript 源码时，看到 `import type` 就知道：这只是一个编译时的类型依赖，不参与程序的实际运行。

## 4.6 重导出（Re-export）

有时候一个文件只是作为"中转站"，把其他模块的内容集中导出去：

```js
// utils/index.js
export { add, subtract } from './math.js';
export { formatDate } from './date.js';
export { capitalize } from './string.js';
```

这样其他文件只需要从一个入口导入：

```js
// main.js
import { add, formatDate, capitalize } from './utils/index.js';
```

不需要关心 `add` 实际定义在 `math.js` 还是 `date.js`。

> 💡 **类比**：重导出就像购物中心的导览台——顾客在一个地方问路，导购员告诉你去哪个楼层（但实际上商品在各层的店铺里）。

### 源码例子：组件目录的索引文件

在 Claude Code 的 `src/components/mcp/index.js` 中（概念转换）：

```js
// src/components/mcp/index.js
export { MCPStdioServerMenu } from './MCPStdioServerMenu.js';
export { MCPToolDetailView } from './MCPToolDetailView.js';
export { MCPToolListView } from './MCPToolListView.js';
```

其他文件只需要：

```js
import { MCPStdioServerMenu } from './components/mcp/index.js';
```

### 先导入再导出

也可以先把内容导入，然后再导出：

```js
// utils.js
import { formatDate, parseDate } from './dateUtils.js';

// ... 使用这些导入做一些事情 ...

// 然后再把它们导出去
export { formatDate, parseDate };
```

这在需要"导入后用一下，同时让外部也能用"的场景很常见。

## 4.7 动态导入

前面讲的 `import` 都是**静态导入**——代码还没运行，导入关系就已经确定了。但有时候你想**按需加载**：需要的时候才去加载某个模块。

这时候用 `import()` 函数：

```js
// 静态导入（一开始就加载）
import { heavyFunction } from './heavyModule.js';

// 动态导入（用到时才加载）
async function doSomething() {
  const { heavyFunction } = await import('./heavyModule.js');
  heavyFunction();
}
```

`import()` 返回一个 **Promise**，所以必须在 `async` 函数里用 `await`，或者用 `.then()`：

```js
// 方式一：async/await
async function loadPlugin() {
  const plugin = await import('./plugins/myPlugin.js');
  plugin.run();
}

// 方式二：.then()
import('./plugins/myPlugin.js').then(plugin => {
  plugin.run();
});
```

### 动态导入的好处

1. **减少初始加载时间**——用户打开页面时不需要加载所有代码
2. **条件加载**——只在需要某个功能时才加载对应的模块
3. **错误处理**——可以用 `try/catch` 捕获加载失败的情况

```js
async function loadOptionalModule() {
  try {
    const module = await import('./optional.js');
    module.init();
  } catch (error) {
    console.log('可选模块加载失败，继续运行');
  }
}
```

### 源码例子：cli.js 中的动态导入

在 Claude Code 的 `src/entrypoints/cli.js` 中（概念转换为纯 JavaScript），有大量的动态导入：

```js
// 来自 src/entrypoints/cli.js（转换为纯 JavaScript）
async function main() {
  const args = process.argv.slice(2);

  // 只在需要时加载性能分析工具
  const { profileCheckpoint } = await import('../utils/startupProfiler.js');
  profileCheckpoint('cli_entry');

  // 如果命令行参数是 --dump-system-prompt，才加载相关模块
  if (args[0] === '--dump-system-prompt') {
    const { enableConfigs } = await import('../utils/config.js');
    enableConfigs();

    const { getSystemPrompt } = await import('../constants/prompts.js');
    const prompt = await getSystemPrompt([]);
    console.log(prompt.join('\n'));
    return;
  }

  // 如果命令行参数是 --daemon，才加载守护进程模块
  if (args[0] === '--daemon') {
    const { daemonMain } = await import('../daemon/main.js');
    await daemonMain(args.slice(1));
    return;
  }

  // ... 更多条件加载
}
```

**逐行拆解**：

1. `await import('../utils/startupProfiler.js')` —— 异步加载 `startupProfiler.js`
2. 返回的对象解构出 `profileCheckpoint` 函数
3. 因为用了 `await`，`main` 函数必须是 `async` 的
4. 每个功能分支只加载自己需要的模块，不会一次加载全部

> 💡 **类比**：动态导入就像点外卖——不是把所有餐厅的食材都囤在家里，而是想吃什么就点什么，送到家再吃。

## 4.8 CommonJS 简要提及

在 ES Module 成为标准之前，Node.js 使用 **CommonJS** 模块系统：

```js
// CommonJS 导出
const PI = 3.14159;
function add(a, b) {
  return a + b;
}
module.exports = { PI, add };

// CommonJS 导入
const math = require('./math.js');
console.log(math.add(2, 3));
```

### ESM vs CommonJS

| | ES Module (ESM) | CommonJS (CJS) |
|---|---|---|
| 导出语法 | `export` / `export default` | `module.exports` |
| 导入语法 | `import` from | `require()` |
| 加载时机 | 静态（编译时） | 动态（运行时） |
| 是否支持顶层 await | ✅ | ❌ |
| 文件扩展名 | `.mjs` 或在 `package.json` 中标记 `"type": "module"` | `.js`（默认） |

Claude Code 使用的是 **ES Module**。你只需知道 CommonJS 存在即可，现代项目越来越多地使用 ESM。

## 本章小结

| 写法 | 含义 | 例子 | Claude Code 使用频率 |
|------|------|------|---------------------|
| `export { foo }` | 命名导出 | `export function foo() {}` | ⭐⭐⭐ 极多 |
| `export default foo` | 默认导出 | `export default function() {}` | ⭐ 很少 |
| `import { foo } from '...'` | 命名导入 | `import { add } from './math.js'` | ⭐⭐⭐ 极多 |
| `import * as X from '...'` | 通配符导入 | `import * as React from 'react'` | ⭐⭐ 较多 |
| `export { foo } from '...'` | 重导出 | `export { foo } from './other.js'` | ⭐⭐ 较多 |
| `export type { Foo }` | 类型重导出（TS 特有，编译后删除） | `export type { Foo } from './other.js'` | ⭐⭐ 较多 |
| `await import('...')` | 动态导入 | `const mod = await import('./mod.js')` | ⭐ 按需加载用 |
| `require()` | CommonJS 导入 | `const fs = require('fs')` | 了解即可 |

**核心原则**：
1. 优先使用命名导出，少用默认导出
2. ESM 导入路径要写 `.js` 后缀
3. 动态导入返回 Promise，需要 `await`
4. Claude Code 主要用 ESM，CommonJS 只需了解
