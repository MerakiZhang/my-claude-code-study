# 第十章：源码实战——追踪 `/clear` 命令的完整链路

> 本章目标：综合运用前面所有知识，追踪用户输入 `/clear` 后，代码是如何一步步执行的。

---

## 10.1 为什么要追踪链路？

读源码最怕"只见树木不见森林"——你知道每个类型是什么意思，但不知道程序怎么跑起来。

**追踪链路**就是选一个具体场景，跟着代码跳转，理解数据是怎么流动的。

我们选的场景是：**用户在终端输入 `/clear`，回车，界面清空。**

---

## 10.2 起点：命令文件

Claude Code 的斜杠命令都放在 `src/commands/` 目录下。找到 `clear`：

### 文件 1：`src/commands/clear/clear.ts`

```ts
import type { LocalCommandCall } from '../../types/command.js'
import { clearConversation } from './conversation.js'

export const call: LocalCommandCall = async (_, context) => {
  await clearConversation(context)
  return { type: 'text', value: '' }
}
```

**逐行解读**：

1. `import type { LocalCommandCall }` → 只导入类型（第四章）
2. `export const call: LocalCommandCall` → `call` 的类型是 `LocalCommandCall`（第二章）
3. `async (_, context) => {...}` → 异步箭头函数（第二章、第六章）
   - `_` → 第一个参数不用，用下划线占位（约定俗成）
   - `context` → 命令执行的上下文
4. `await clearConversation(context)` → 等待清空逻辑完成（第六章）
5. `return { type: 'text', value: '' }` → 返回一个空文本结果

**问题**：`LocalCommandCall` 到底是什么类型？

---

## 10.3 往上追踪：LocalCommandCall 的类型定义

```ts
// 来自 src/types/command.ts（根据导入路径推测）
type LocalCommandCall = (
  args: string,
  context: CommandContext,
) => Promise<CommandResult>
```

所以 `clear.ts` 中的 `call` 函数必须满足：
- 接收两个参数：`args`（字符串）和 `context`（CommandContext）
- 返回 `Promise<CommandResult>`

`clear.ts` 完全符合：
- `async (_, context)` → 两个参数 ✅
- `return { type: 'text', value: '' }` → `CommandResult` 的一种 ✅

## 10.4 往下追踪：clearConversation 的实现

```ts
// 来自 src/commands/clear/conversation.ts
export async function clearConversation({
  setMessages,
  readFileState,
  discoveredSkillNames,
  loadedNestedMemoryPaths,
  getAppState,
  setAppState,
  setConversationId,
}: {
  setMessages: (updater: (prev: Message[]) => Message[]) => void
  readFileState: FileStateCache
  discoveredSkillNames?: Set<string>
  loadedNestedMemoryPaths?: Set<string>
  getAppState?: () => AppState
  setAppState?: (f: (prev: AppState) => AppState) => void
  setConversationId?: (id: UUID) => void
}): Promise<void> {
```

**注意这个参数写法**！它不是接收多个独立参数，而是接收一个对象，然后立即解构（第二章）。

等价于：

```ts
export async function clearConversation(params: {
  setMessages: ...
  readFileState: ...
  // ...
}): Promise<void> {
  const { setMessages, readFileState, ... } = params;
  // ...
}
```

### 参数类型分析

| 参数 | 类型 | 是否可选 |
|------|------|---------|
| `setMessages` | `(updater: (prev: Message[]) => Message[]) => void` | 必填 |
| `readFileState` | `FileStateCache` | 必填 |
| `discoveredSkillNames` | `Set<string>` | 可选 (`?`) |
| `loadedNestedMemoryPaths` | `Set<string>` | 可选 |
| `getAppState` | `() => AppState` | 可选 |
| `setAppState` | `(f: ...) => void` | 可选 |
| `setConversationId` | `(id: UUID) => void` | 可选 |

这些可选参数的存在说明 `clearConversation` 可以被不同场景调用——有些场景可能没有完整的上下文。

## 10.5 继续追踪：clearConversation 做了什么

这个函数有 251 行，我们不需要读完，只需要理解它的**骨架**：

### 阶段 1：执行 SessionEnd hooks（第 68-74 行）

```ts
const sessionEndTimeoutMs = getSessionEndHookTimeoutMs()
await executeSessionEndHooks('clear', {
  getAppState,
  setAppState,
  signal: AbortSignal.timeout(sessionEndTimeoutMs),
  timeoutMs: sessionEndTimeoutMs,
})
```

- `await` → 等待异步操作完成（第六章）
- `AbortSignal.timeout(...)` → 设置超时，防止 hook 卡住

### 阶段 2：清理消息（第 109 行）

```ts
setMessages(() => [])
```

把消息数组设为空。这就是为什么界面被清空了！

### 阶段 3：清理状态（第 127-191 行）

```ts
if (setAppState) {
  setAppState(prev => {
    // ... 大量清理逻辑
    return {
      ...prev,
      tasks: nextTasks,
      attribution: createEmptyAttributionState(),
      // ...
    }
  })
}
```

这里用了 `setAppState` 的函数式更新模式（`prev => newState`），这正是我们在 `Task.ts` 中看到的 `SetAppState` 类型：

```ts
type SetAppState = (f: (prev: AppState) => AppState) => void
```

### 阶段 4：生成新 Session ID（第 203 行）

```ts
regenerateSessionId({ setCurrentAsParent: true })
```

清空的会话获得新 ID，用于分析追踪。

### 阶段 5：执行 SessionStart hooks（第 245-250 行）

```ts
const hookMessages = await processSessionStartHooks('clear')
if (hookMessages.length > 0) {
  setMessages(() => hookMessages)
}
```

如果有启动 hook 产生了消息，就显示出来。

## 10.6 类型收窄的实际应用

`conversation.ts` 中有大量类型收窄的例子。看这一段（第 95-106 行）：

```ts
const shouldKillTask = (task: AppState['tasks'][string]): boolean =>
  'isBackgrounded' in task && task.isBackgrounded === false

if (getAppState) {
  for (const task of Object.values(getAppState().tasks)) {
    if (shouldKillTask(task)) continue
    if (isLocalAgentTask(task)) {
      preservedAgentIds.add(task.agentId)
      preservedLocalAgents.push(task)
    } else if (isInProcessTeammateTask(task)) {
      preservedAgentIds.add(task.identity.agentId)
    }
  }
}
```

**知识点回顾**：
- `'isBackgrounded' in task` → `in` 操作符收窄（第三章）
- `isLocalAgentTask(task)` → 自定义类型守卫（第三章）

`isLocalAgentTask` 的返回类型是 `task is LocalAgentTaskState`，所以当它返回 `true` 时，TypeScript 自动把 `task` 收窄为 `LocalAgentTaskState`，于是可以安全访问 `task.agentId`。

再看第 155-159 行：

```ts
if ('abortController' in task) {
  task.abortController?.abort()
}
if ('unregisterCleanup' in task) {
  task.unregisterCleanup?.()
}
```

`'abortController' in task` 先检查属性是否存在，然后 TypeScript 才允许访问 `task.abortController`。这就是类型收窄保护你不会访问不存在的属性。

## 10.7 链路图

把 `/clear` 的执行链路画出来：

```
用户输入 /clear
    ↓
命令解析器路由到 src/commands/clear/clear.ts
    ↓
clear.ts::call() 被调用
    ↓
await clearConversation(context)
    ↓
conversation.ts::clearConversation()
    ├── 1. 执行 SessionEnd hooks
    ├── 2. setMessages(() => [])     ← 界面清空！
    ├── 3. setAppState(prev => {...}) ← 清理状态
    ├── 4. regenerateSessionId()      ← 新会话ID
    └── 5. 执行 SessionStart hooks
    ↓
返回 { type: 'text', value: '' }
    ↓
界面显示空结果（实际上就是什么都不显示）
```

## 10.8 本章小结

通过这个链路追踪，你学会了：

1. **从命令文件入手** → `src/commands/<name>/<name>.ts`
2. **读函数签名理解输入输出** → `LocalCommandCall` 类型
3. **追踪异步调用** → `await clearConversation(context)`
4. **识别状态更新模式** → `setAppState(prev => ...)`
5. **发现类型收窄的实际作用** → `isLocalAgentTask(task)`
